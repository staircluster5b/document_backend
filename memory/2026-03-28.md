## Nội dung chính đã nghiên cứu

1. **Bản chất Service Discovery**: Database phân tán eventually consistent, giải quyết vấn đề ephemeral IPs trong microservices

2. **Hai mô hình**:
   - Client-side: Client tự query registry và load balance (Eureka, Consul)
   - Server-side: Load balancer làm proxy (K8s DNS + kube-proxy)

3. **So sánh 3 giải pháp**:
   - **Eureka**: Client-side caching, self-preservation mode, Spring Cloud native
   - **Consul**: Raft consensus, multi-DC, service mesh (Connect + Envoy)
   - **K8s DNS**: DNS-based, server-side LB, Kubernetes native

4. **Trade-offs quan trọng**:
   - CAP: Tất cả chọn AP (Availability + Partition tolerance)
   - Client-side: Complexity ở client nhưng linh hoạt
   - Server-side: Đơn giản nhưng phụ thuộc LB

5. **Rủi ro production**:
   - Thundering herd khi cold start
   - Split-brain trong Eureka
   - DNS caching issues (Java mặc định cache forever)
   - False positive health checks
